== Part 1 — Deploying a containerized app with OpenShift Build

=== Know

_Before we follow a developer through their inner loop, let's see how OpenShift handles application deployment as a platform capability. OpenShift Build runs container builds inside the cluster from a source repository. With the **Docker strategy**, the platform reads a Dockerfile (or Containerfile) from the repo, builds an image, pushes it to the internal registry, and tracks it with an `ImageStream`. No external CI system, no developer workstation builds, and no pre-built image registry needed — the platform handles everything from source to running container._

**Business challenge:**

* Developers wait for centralized build infrastructure to take a Git repo to a running container
* Container builds running on developer workstations create environment drift and inconsistent images
* Image provenance is unclear without a controlled, in-cluster build process

**Value proposition:**

OpenShift Build executes container builds inside the cluster, governed by the platform. Every build is reproducible, auditable, and produces an `ImageStream` that downstream deployments and pipelines can reference. The Docker strategy is the right fit when the application repo already provides a container build recipe (a `Dockerfile` or `Containerfile`).

=== Show

**Steps:**

. Open the **OpenShift web console** at {console_url}[{console_url}^] and log in as `{dev_user}` / `{password}`

. Open the **Web Terminal** by clicking the `>_` icon in the top navigation bar (accept the defaults if prompted) and create a new project for the deployment scenarios:
+
[source,bash,role="copypaste",subs="attributes+"]
----
oc new-project {nodejs_demo_ns}
----
+
* "We'll use this project for both deployment scenarios. Creating it from the CLI is one command — same result as the web-console flow."

. Switch to the **Developer** perspective and confirm `{nodejs_demo_ns}` is selected in the project dropdown

. Click **+Add** in the left sidebar and select **Import from Git**

. Enter the repository URL:
+
[source,text,role="copypaste",subs="attributes+"]
----
{nodejs_app_repo}
----

. Under **Import Strategy**, select **Dockerfile**:
* The console scans the repo and confirms a `Dockerfile` is present
* "OpenShift inspects the repo, finds the Dockerfile, and proposes a Docker-strategy build."

. Set **Application name** to `nodejs-rest-http-docker` and **Name** to `nodejs-rest-http-docker`

. Leave the default resource type (`Deployment`) and click **Create**

. Watch the build complete in the **Topology** view:
* The pod cycles through `Build → Push → Deploy → Running` (the blue donut fills as the build progresses)
* "OpenShift just cloned the repo, built the image inside the cluster, pushed it to the internal registry, and rolled out a Deployment — all from a single form."

. Inspect the platform-managed artifacts:
.. Navigate to **Builds → BuildConfigs**, open `nodejs-rest-http-docker`, and show the `strategy.dockerStrategy` block in the YAML
.. Navigate to **Builds → ImageStreams** and show the new ImageStream tracking the built image
.. "These are platform-managed resources. The image lives in the in-cluster registry, the BuildConfig is the reusable recipe, and the ImageStream provides a stable reference for downstream consumers."

. Open the route on the Topology view and confirm the Node.js REST endpoint responds

**What to notice:**

* The audience went from a Git URL to a running container in a single form, with no external CI
* The build executed inside the cluster, producing in-cluster artifacts (BuildConfig, ImageStream, image in the registry)
* No image was pre-built, no Dockerfile rebuild on a workstation, and no separate image push step required

**Business value:**

"Every container build is governed by the platform, every image lives in the in-cluster registry with a stable ImageStream reference, and every deployment is traceable to a specific BuildConfig and source revision. This is the foundational deployment path before any pipelines are layered on top."

== Part 2 — Deploying source code with Source-to-Image (S2I)

=== Know

_Many applications do not provide their own Dockerfile. With **Source-to-Image (S2I)**, OpenShift detects the application's language, picks a Red Hat-maintained builder image, and produces a runnable container without the developer writing any container build configuration. The same `BuildConfig` + `ImageStream` artifacts result — only the build strategy changes._

**Business challenge:**

* Application teams without container expertise cannot ship to OpenShift on their own
* Hand-maintained Dockerfiles drift from organizational base image and security standards
* Onboarding a new application requires authoring and reviewing container build files

**Value proposition:**

S2I separates application code from container build authoring. Red Hat ships hardened, supported builder images for common runtimes (Node.js, Java, Python, .NET, Go, Ruby). The platform picks the right one, layers the application on top, and produces a consistent container — without any Dockerfile in the repo.

=== Show

**Steps:**

. In the **OpenShift web console** Developer perspective, in the `{nodejs_demo_ns}` project, click **+Add** and select **Import from Git** again

. Enter the same repository URL:
+
[source,text,role="copypaste",subs="attributes+"]
----
{nodejs_app_repo}
----

. Under **Import Strategy**, select **Builder Image**:
* The console auto-detects Node.js as the language and proposes the Red Hat Node.js builder image
* "OpenShift's S2I strategy recognized this as a Node.js project, picked the Red Hat-maintained Node.js builder image, and configured a Source-strategy BuildConfig — without us editing anything."

. Set **Application name** to `nodejs-rest-http-s2i` and **Name** to `nodejs-rest-http-s2i`

. Confirm the suggested **Builder Image version** (latest supported Node.js)

. Leave the default resource type (`Deployment`) and click **Create**

. Watch the build complete in the **Topology** view alongside the previous `nodejs-rest-http-docker` deployment

. Inspect the resulting BuildConfig:
.. Navigate to **Builds → BuildConfigs** and open `nodejs-rest-http-s2i`
.. Show `strategy.sourceStrategy` referencing the Node.js builder image — contrast with the `dockerStrategy` from Part 1
.. "Same source repo, same platform mechanics, different strategy. The Source strategy uses a Red Hat builder image; the Docker strategy uses the repo's own Dockerfile."

. Open the route for the S2I-built app and confirm the same Node.js REST endpoint responds

. _After demonstrating both Part 1 and Part 2_, clean up the demo project to remove the BuildConfigs, ImageStreams, Deployments, and routes created in this section:
+
[source,bash,role="copypaste",subs="attributes+"]
----
oc delete project {nodejs_demo_ns}
----
+
* "One command tears down everything the two scenarios created. The remaining modules use the Parasol application namespaces, not this sandbox."

**What to notice:**

* The audience deployed the **same source repo** twice with two different strategies, side by side
* S2I needed no Dockerfile and no container build expertise from the developer
* Both deployments produce identical platform artifacts (BuildConfig, ImageStream, Deployment, Service, Route) — only the build strategy differs
* Red Hat builder images are versioned, supported, and updated centrally

**Business value:**

"S2I lets application teams ship to OpenShift without becoming container experts. Builder images are maintained by Red Hat — they get the security patches, the runtime updates, and the base-image hardening for free. Combined with the Docker strategy from Part 1, the platform supports any team's deployment style: bring your own Dockerfile, or hand the platform your source code."

* Developers maintain their own tooling, creating security blind spots and configuration drift
* Teams waste hours each sprint troubleshooting environment-specific issues

**Current state at Parasol:**

* 15-20 hours per developer per quarter lost to environment troubleshooting

== Part 4 — Adding a feature with Quarkus dev mode

=== Know

_Our new developer is now in their cloud workspace and ready to contribute. But first, let them explore the application. Parasol Insurance is a claims and customer management platform used by Parasol's agents to process insurance claims and communicate with customers. It uses enterprise messaging with Apache Kafka to process incoming emails through a keyword matching algorithm that routes them to the appropriate department. (This routing will be enhanced with AI in Section 3.) The developer has been assigned their first task: add a new claims statistics API endpoint. Using Quarkus dev mode, they can write code and see changes reflected immediately without manual builds or restarts. However, in their eagerness to ship, the developer takes some shortcuts — using `System.out.println` instead of proper logging and wrapping code in an empty catch block — code smells that will have consequences in Module 2._

**Business challenge:**

* Inner loop (code-build-test) averages 20-30 minutes per iteration
* Developers lose flow state waiting for builds to complete
* Slow feedback loops discourage experimentation and innovation
* Code quality issues slip through when developers prioritize speed over best practices

**Current state at Parasol:**

* Average developer makes 8-12 code changes per day, each requiring a full rebuild
* 3-5 hours per developer per day spent waiting for build and test cycles
* No automated code quality enforcement, so shortcuts like improper logging and swallowed exceptions accumulate as technical debt
* Defects and maintenance burden grow as code smells compound across the codebase

**Value proposition:**

Quarkus dev mode provides live reload and continuous testing capabilities that reduce the inner loop from minutes to seconds. Developers see their changes reflected immediately, and the framework supports instant feedback on code changes. This keeps developers in flow state and dramatically accelerates feature delivery.

. Create the new claims statistics endpoint. In the Explorer on the left, right-click on the `java / com / Parasol` directory and select **New File**. Name the file `ClaimsStatsResource.java`. Then copy and paste the following code into the new file:
+
[source,java,role="copypaste"]
----
package com.parasol;

import java.util.HashMap;
import java.util.Map;

import jakarta.inject.Inject;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;

@Path("/api/claims/stats")
@Produces(MediaType.APPLICATION_JSON)
public class ClaimsStatsResource {

	@Inject
	ClaimsResource claimsResource;

	@GET
	public Map<String, Object> getStats() {
		Map<String, Object> stats = new HashMap<>();

		System.out.println("Fetching claim statistics");

		try {
			var claims = claimsResource.getAllClaims();
			stats.put("total", claims.size());

			Map<String, Long> byCategory = new HashMap<>();
			for (var claim : claims) {
				byCategory.merge(claim.getCategory(), 1L, Long::sum);
			}
			stats.put("by_category", byCategory);

			Map<String, Long> byStatus = new HashMap<>();
			for (var claim : claims) {
				byStatus.merge(claim.getStatus(), 1L, Long::sum);
			}
			stats.put("by_status", byStatus);
		} catch (Exception e) {
		}

		return stats;
	}
}
----

* Upon accessing the app again, the application reloads automatically (no restart required)
* In the browser address bar, replace the `dashboard.html` with `api/claims/stats` and hit enter to access the new endpoint
* Open `/api/claims/stats` in the browser preview to show the new endpoint returning claim statistics
* No manual build step, no repackaging, no redeployment

image::parasol-claims-stats-devmode.png[Browser showing the /api/claims/stats endpoint returning JSON claim statistics data in Quarkus dev mode,link=self,window=blank,align="center",width=700,title="Claims statistics endpoint working in dev mode"]

* No centralized governance over IDE configurations, extensions, or security policies
* Inconsistent environments across teams lead to integration failures and security blind spots
* Scaling development environments manually does not keep pace with organizational growth
